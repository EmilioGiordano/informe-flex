1. Anatomía de un archivo .l

Toda especificación de Flex (y de Lex, y de casi toda la familia) tiene tres secciones separadas por %%:

DEFINICIONES atajos de expresiones regulares (ej. DIGITO [0-9]) %% REGLAS patrón → acción (qué hacer al reconocerlo) %% CÓDIGO DE USUARIO funciones auxiliares, main, etc.
La sección del medio (las reglas) es el corazón. Cada regla es <expresión regular> { acción }: cuando el patrón matchea, se ejecuta la acción (típicamente, devolver o imprimir un token).

2. Un ejemplo real

Este archivo lexer.l reconoce un mini-lenguaje aritmético: números, identificadores y los operadores + - * / = ( ).

%{ #include <stdio.h> %} DIGITO [0-9] LETRA [a-zA-Z_] %% {DIGITO}+ { printf("NUMERO(%s)\n", yytext); } {LETRA}({LETRA}|{DIGITO})* { printf("IDENTIFICADOR(%s)\n", yytext); } "+" { printf("MAS\n"); } "-" { printf("MENOS\n"); } "*" { printf("POR\n"); } "/" { printf("DIV\n"); } "=" { printf("ASIGNACION\n"); } "(" { printf("PAR_IZQ\n"); } ")" { printf("PAR_DER\n"); } [ \t\n]+ { /* espacios: se ignoran */ } . { printf("DESCONOCIDO(%s)\n", yytext); } %% int main(void) { yylex(); return 0; } int yywrap(void) { return 1; }
Cómo leerlo

Arriba, las definiciones (DIGITO, LETRA) son atajos. En las reglas, cada expresión regular tiene su acción: {DIGITO}+ imprime un NUMERO, etc. La regla . (cualquier carácter) al final atrapa lo inesperado. En el código, main llama a yylex(), la función que Flex genera.

3. Ejecutarlo: tres comandos

El flujo completo, de la especificación a un programa que corre:

flex lexer.l # genera "lex.yy.c": el analizador en C gcc lex.yy.c -o lexer # compila ese analizador echo "a = 4 + 2" | ./lexer
lexer.ltus reglas →flex lex.yy.cautómata en C →gcc lexerejecutable → tokensla salida

Con la entrada a = 4 + 2 (el mismo ejemplo del PDF de la cátedra), la salida real es:

IDENTIFICADOR(a) ASIGNACION NUMERO(4) MAS NUMERO(2)
Otra entrada, más rica

Con resultado = (x1 + 23) * 2:

IDENTIFICADOR(resultado) ASIGNACION PAR_IZQ IDENTIFICADOR(x1) MAS NUMERO(23) PAR_DER POR NUMERO(2)

Fijate que x1 salió como un identificador (no x+1) y 23 como un número: el match más largo (Unidad 2) funcionando solo.

4. Lo que Flex generó (el momento wow)

Acá está la esencia del tema, hecha tangible. Vos escribiste un puñado de reglas; Flex escribió un analizador completo:

ArchivoQué esLíneas
lexer.ltu especificación (lo que escribís)37
lex.yy.cel analizador léxico completo que generó Flex1813
El "wow"

De 37 líneas de expresiones regulares, Flex generó 1813 líneas de C que implementan un autómata finito. Ese es el significado literal de "generador": el paso del medio (flex) escribió por vos el código que a mano sería largo y tedioso. Qué contiene ese código generado lo desarmamos en la Unidad 5.

5. Las variables y funciones que da Flex

Flex pone a tu disposición un puñado de identificadores (todos empiezan con yy):

NombreQué es
yylex()La función del analizador, generada automáticamente. La llamás para que produzca tokens.
yytextEl lexema: el texto exacto que se acaba de reconocer.
yyin / yyoutDe dónde lee / a dónde escribe (por defecto, entrada y salida estándar).
yywrap()Qué hacer al llegar al fin de archivo (devolver 1 = "terminé").
yylinenoEl número de línea actual (útil para mensajes de error).
Probalo vos

Si tenés Flex instalado (en Linux/WSL: sudo apt install flex), el archivo ejemplos/lexer.l de este proyecto es exactamente el de arriba. Corré los tres comandos y vas a ver los tokens en tu pantalla. Es la mejor demo posible: el tema funcionando.
